# Mission Control Dashboard — Proposal

**Author:** Subagent research task  
**Date:** 2026-03-31  
**Context:** Visibility layer for Victor's multi-agent OpenClaw setup (Ceil + Neo)

---

## 1. What "Mission Control" Means Here

Victor runs two top-level agents on OpenClaw:

| Agent | Role | Channel |
|-------|------|---------|
| **Ceil** (`agent:main`) | Primary assistant — operations, research, ProTek FSM | Telegram |
| **Neo** (`agent:neo`) | Cybersecurity / HTB | Telegram |

Both agents **spawn subagents** for delegated tasks. Victor currently has **no unified view** of what's running, what's pending, or what just finished — he checks manually or waits for completion messages.

The goal: a lightweight, self-hosted dashboard that gives Victor **at-a-glance situational awareness** of his entire agent fleet.

## 3. Data Sources — What OpenClaw Gives Us

All data lives in `~/.openclaw/`. Everything is plain JSON or newline-delimited JSON (JSONL). No proprietary DB.

  updatedAt: 1774923079569,        ← Unix ms — last activity

**Access:** Direct file read + `openclaw sessions list` CLI.

### 3.3 Session Message Log — "What Did It Say/Do"
**File:** `~/.openclaw/agents/{agent}/sessions/{sessionId}.jsonl`

Each line is a JSON object with a `type` field:

| type | Contents |
|------|----------|
| `session` | Session metadata (version, id, cwd) |
| `message` | User or assistant message (role, content, usage stats) |
| `thinking_level_change` | Thinking level toggled |
| `custom` / `model-snapshot` | Model selection events |
| `tool_use` / `tool_result` | Tool invocations |
| `task_input` / `task_output` | Subagent task I/O |

**What it tells you:** The full conversation transcript, tool calls made, tokens used.

**Access:** Stream via `tail -f` or read last N lines. Can be sampled for recent activity.

## 4. Architecture

### 4.1 Recommended: Node.js Polling Backend + Vanilla HTML Frontend

This matches the existing protek-dashboard pattern exactly.

```
┌─────────────────────────────────────────────────────────────┐
│                        VPS (Hostinger)                       │
│                                                             │
│  ┌──────────────┐    ┌─────────────┐    ┌────────────────┐  │
│  │  OpenClaw    │    │  Node.js    │    │   Browser      │  │
│  │  data files  │───▶│  Backend    │◀───│   Dashboard   │  │
│  │  (JSON/JSONL)│    │  (polling)  │    │   (HTML/JS)    │  │
│  └──────────────┘    └──────┬──────┘    └────────────────┘  │
│                             │                                │
│                      SSE / HTTP polling                      │
└─────────────────────────────────────────────────────────────┘
```

**Why this over alternatives:**

| Option | Verdict | Reason |
|--------|---------|--------|
| Web dashboard (this) | ✅ **Recommended** | Victor already has one, proven pattern, browser-accessible |
| CLI/TUI (blessed, htop-style) | ❌ Not for Victor | He wants visual dashboards, already has web habit |
| Integrate with existing Protek dashboard | ⚠️ Consider later | Keep separate for now, merge later if it makes sense |
| WebSocket direct to OpenClaw | ❌ Not possible | OpenClaw doesn't expose a WS API |
| SSE from OpenClaw gateway | ❌ Not needed | Simple polling is sufficient for this use case |

### 4.2 Backend: Node.js Express

- Polls data files every **5 seconds** (configurable)
- Exposes REST endpoints + SSE stream
- Serves the static HTML frontend
- Zero external dependencies beyond Express + `tail` pipe

**Endpoints:**

| Method | Path | Returns |
|--------|------|---------|
| GET | `/api/agents` | Session status for all agents |
| GET | `/api/subagents` | Recent subagent runs (last 50) |
| GET | `/api/cron` | Cron jobs with state |
| GET | `/api/activity` | Last N lines from active session JSONL |
| GET | `/api/system` | `ps aux`, uptime, memory |
| GET | `/api/stream` | SSE stream — pushes all of the above on change |

### 4.3 Real-Time: Server-Sent Events (SSE)

SSE is simpler than WebSocket — it's one-way server→client, which is all Victor needs. The Node backend polls files, detects changes, and pushes a JSON event to all connected browsers.

**Client reconnect is automatic in browsers.** Victor just opens the page and watches.

## 5. Tech Stack

| Layer | Choice | Rationale |
|-------|--------|-----------|
| Frontend | Vanilla HTML + Tailwind CDN + Lucide icons | Matches protek-dashboard; no build step |
| Charts | Chart.js (CDN) | Already used in protek-dashboard |
| Backend | Node.js (plain, no framework) | Victor knows this; fits self-hosted |

| Real-time | SSE | One-way, browser-native, low complexity |

## 9. Critical Design Decisions

### 9.1 Keep It Self-Contained
The dashboard should be **one Node.js server + one HTML file**. No database, no Redis, no external services. Everything derived from OpenClaw's existing files on disk.

### 9.2 Polling Over Push
OpenClaw doesn't have a WebSocket event emitter for file changes. Simple polling (5s interval) is sufficient — Victor doesn't need millisecond-accurate real-time. SSE is used only to avoid the overhead of HTTP connection setup per poll.

### 9.3 Merge With Protek Dashboard?
**No — keep them separate.** The Protek dashboard is operations-specific; the mission control dashboard is infrastructure/agent-specific. They have different audiences (Victor checking ProTek metrics vs Victor checking agent health). Merge only if Victor specifically asks.

### 9.4 Who Is This For?
This dashboard is **Victor only**. No auth needed if accessed via VPN/localhost. If exposed externally, add basic HTTP auth or token-based access.

### 9.5 Alerting
The dashboard is **visibility, not alerting** — though it naturally surfaces problems (cron errors, subagent failures). Future: could add Telegram notifications when `consecutiveErrors` increases, but that's Phase 3+.
