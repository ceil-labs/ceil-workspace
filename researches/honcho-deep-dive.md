# Honcho Deep Dive: Alternative to DIY Karpathy Memory

**Date:** 2026-04-04  
**Context:** Re-evaluating Honcho as a serious alternative after DIY Karpathy was identified as most promising among Hindsight, OpenViking, ByteRover, and Memary.

---

## 1. What Honcho Actually Is

Honcho is a **peer-centric, reasoning-first memory infrastructure** developed by [Plastic Labs](https://plasticlabs.ai). It builds and maintains rich, evolving representations ("peer representations") of users and agents by **continually reasoning about conversations** rather than just storing and retrieving them.

This is fundamentally different from the DIY Karpathy approach, which treats memory as document/wiki storage that an LLM reads and synthesizes on-demand. Honcho instead runs background inference over your conversation history to proactively extract conclusions, patterns, and deductive insights — then makes those queryable at inference time.

**Official site:** https://honcho.dev  
**GitHub:** https://github.com/plastic-labs/honcho  
**Documentation:** https://docs.honcho.dev/v3/

## 2. Core Concepts

### 2.1 Peer Representations

Everything in Honcho revolves around **peers** — representations of any entity (human user, AI agent, NPC, etc.) that persist and deepen over time. A peer is not just a user ID; it's a rich, reasoning-produced model of that entity built from all their messages across all sessions.

Each peer representation contains:
- **Explicit conclusions** — things explicitly stated ("user prefers Python")
- **Deductive conclusions** — inferences drawn from premises ("user is likely a developer")
- **Inductive conclusions** — patterns recognized across multiple messages
- **Abductive conclusions** — simplest explanations for observed behavior
- **Peer cards** — structured biographical summaries
- **Summaries** — compressed session histories

The reasoning is powered by formal logic frameworks, not just statistical co-occurrence. This means Honcho can infer things that were never explicitly stated but logically follow from what was said.

### 2.2 Dialectic Reasoning (The "Dialectic" Worker)

The **Dialectic** is one of Honcho's four background workers. When you query Honcho about a user ("what does the user want?"), Dialectic answers with 5 depth levels:

| Depth | When Used | Model |
|-------|-----------|-------|
| minimal | Quick facts | GLM-4.7 Flash |
| low | Per-turn context | GLM-4.7 Flash |
| medium | Complex queries | Grok-4.1 Fast |
| high | Multi-hop reasoning | GLM-5 |
| max | Hardest synthesis | GLM-5 |

Dialectic answers questions by retrieving relevant conclusions from vector storage and reasoning over them with increasing rigor at higher depths. It uses different LLM providers per depth tier (light models for quick answers, heavy models for complex synthesis).

**Critical note:** The custom Neuromancer XR model (86.9% on LoCoMo benchmark vs 69.6% for base Qwen3-8B and 80.0% for Claude 4 Sonnet) is the specialized model designed for this logical conclusion extraction. It's trained on ~10,000 curated social reasoning traces and follows formal logic rules rather than generating plausible-sounding text.

### 2.3 The Four Workers

Honcho runs four async background workers that process your conversation data:

| Worker | What It Does | Frequency | Model |
|--------|-------------|-----------|-------|
| **Deriver** | Extracts observations/conclusions from messages | Every message (batched at ~1000 tokens) | GLM-4.7 Flash |
| **Dialectic** | Answers questions about the user on demand | On-demand, 5 depth levels | Tiered GLM-4.7 → Grok-4.1 → GLM-5 |
| **Summary** | Compresses long sessions into summaries | Every 20/60 messages | GLM-4.7 Flash |
| **Dream** | Memory consolidation — merges redundant/contradictory conclusions, infers patterns | Every ~8 hours | GLM-5 |

The **Deriver** is the most important for memory formation. It reads every message and extracts explicit and deductive conclusions. These are stored as structured JSON with explicit premises and deductive conclusions.

### 2.4 Peer Cards

Peer cards are structured biographical summaries about a peer. They represent key facts about an entity — preferences, communication style, goals, constraints, relationships. Peer cards are generated by the Deriver and updated over time as new information arrives.

In multi-agent scenarios, peer cards enable **agent-to-agent memory**:
- Agent A can see Agent B's peer card
- This allows agents to understand each other's capabilities, personality, and state
- The OpenClaw plugin creates a separate peer for each agent (`agent-{id}`)

### 2.5 Profile Scoping & Cross-Session Memory

Honcho's memory model is inherently cross-session. Every message a peer sends contributes to their peer representation, regardless of which session it came from. Sessions are just temporal containers — the peer representation accumulates knowledge across all of them.

This is fundamentally different from DIY Karpathy, where cross-session memory requires an explicit LLM compilation step (`compile.py`) that synthesizes raw messages into a `wiki/` document.

**Scoping works hierarchically:**
- **Workspace level** — complete isolation between applications
- **Peer level** — per-entity memory containers
- **Session level** — temporal context, message ordering

### 2.6 Session Management

Sessions are lightweight interaction threads. They can be:
- Multi-peer (group conversations)
- Single-peer (for ingesting external data as messages)
- Scoped with perspective controls (whether a peer observes others in the session)

The OpenClaw plugin automatically creates sessions for each conversation context and tracks parent→subagent relationships via `parentPeerId` metadata.

## 3. Architecture

### 3.1 Data Model

```
Workspace (namespace isolation)
├── Peers (users, agents — peer representations)
│   └── PeerCards, Conclusions, Summaries
├── Sessions (interaction threads)
│   └── Messages (fundamental units)
└── Vector Collections (indexed conclusions)
```

**Key entities:**
- **Workspace** — top-level container, provides complete isolation
- **Peer** — any entity with a persistent representation
- **Session** — temporal container for messages between peers
- **Message** — attributed content that triggers reasoning

### 3.2 Storage Backend

- **PostgreSQL 15 + pgvector** — primary storage for messages, peers, sessions, conclusions, vector embeddings
- **Redis 8.2** — caching and async job queue (queues for Deriver, Summary, Dream workers)
- **Vector embeddings** — OpenAI `text-embedding-3-small` (or custom endpoint)

### 3.3 API Design

- **REST API** — full CRUD over workspaces, peers, sessions, messages
- **WebSocket** — real-time updates (future)
- **Chat endpoint** — `POST /v3/chat` — primary agent integration point; takes messages, returns reasoning-enhanced context
- **Get Context endpoint** — retrieves relevant conclusions + recent messages for prompt injection

### 3.4 Cloud vs. Self-Hosted

| | Managed Cloud (honcho.dev) | Self-Hosted |
|-|---------------------------|-------------|
| Data storage | Plastic Labs servers | Your infrastructure |
| LLM inference | Neuromancer XR (custom) | Any OpenAI-compatible API |
| Neuromancer XR | ✅ Available | ❌ Not available (general-purpose models instead) |
| Setup | Zero | ~3 min with community fork |
| Cost | Free tier / paid | API usage + infrastructure |
| Maintenance | None | Yours |

## 4. Self-Hosted Options

### 4.1 Official Self-Hosting (from upstream Honcho)

**Documentation:** https://docs.honcho.dev/v3/contributing/self-hosting

Official self-hosting uses Docker Compose:

```yaml
# docker-compose.yml.example (upstream)
services:
  api:
    image: honcho:latest
    ports: ["8000:8000"]
    depends_on: [database, redis]
    # FastAPI server + entrypoint script
  deriver:
    # Background worker — processes messages
  database:
    image: pgvector/pgvector:pg15
    ports: ["5432:5432"]
  redis:
    image: redis:8.2
    ports: ["6379:6379"]
  prometheus:
  grafana:
```

**Requirements:**
- Docker Engine + Compose plugin
- PostgreSQL 15 with pgvector extension
- Redis 8.2
- Any OpenAI-compatible LLM API key
- uv (for Python package management)

**What's NOT included in the official docker-compose:**
- No pre-configured LLM providers (you must set up env vars)
- No automatic startup script
- The 3 additional workers (Dialectic, Summary, Dream) are mentioned but not all included in the base Docker setup

**Setup steps (official):**
```bash
git clone https://github.com/plastic-labs/honcho.git
cp .env.template .env
# Edit .env with API keys
cp docker-compose.yml.example docker-compose.yml
docker compose up -d
```

### 4.2 Community Fork: elkimek/honcho-self-hosted

**Repository:** https://github.com/elkimek/honcho-self-hosted  
**Purpose:** Specifically built for Hermes Agent (Nous Research) but works for any Honcho deployment  
**Maturity:** Active, documented, production-capable

This is the most practical self-hosted option. It's not a fork of Honcho code — it's a **3-file overlay** on top of upstream Honcho (docker-compose.yml + config.toml + env template) plus a setup script.

**What it adds:**
1. `setup.sh` — one-command installer (~3 minutes end-to-end)
2. `config.toml` — pre-configured LLM providers with primary + backup failovers
3. Hardened docker-compose with proper user `honcho` (not `postgres`)
4. Ports bound to 127.0.0.1 only (security)
5. Prometheus + Grafana removed (simpler)

**Setup (3 minutes):**
```bash
curl -sL https://raw.githubusercontent.com/elkimek/honcho-self-hosted/main/setup.sh | bash
```
This script:
- Installs Docker if needed
- Clones Honcho upstream
- Copies config files
- Prompts for API keys
- Starts everything
- Optionally configures Hermes Agent

**LLM Provider Configuration:**
```toml
# config.toml
[deriver]
PROVIDER = "vllm"      # primary slot
MODEL = "z-ai/glm-4.7-flash"
BASE_URL = "https://openrouter.ai/api/v1"
API_KEY = "${LLM_VLLM_API_KEY}"

[deriver.backup]
PROVIDER = "custom"
MODEL = "..."
```

Supports: OpenRouter, Venice, Together, Groq, Ollama (local), vLLM (local), OpenAI direct, Anthropic direct, Google Gemini.

### 4.3 Self-Hosted: Key Tradeoffs

**If you self-host, you lose:**
- Neuromancer XR (custom reasoning model) — must use general-purpose models instead
- Plastic Labs' managed infrastructure and SLAs
- Zero-maintenance operation

**If you self-host, you gain:**
- Data sovereignty (all conversation data on your machine)
- Custom LLM providers (OpenRouter, Venice, etc.)
- Potentially lower cost at scale
- Full control over retention

**On custom models for self-hosted:** Honcho's docs acknowledge that general-purpose models work well for observation extraction, and capable models (GLM-5, Grok 4.1) with strong function calling largely close the gap. The main advantage of self-hosting is data sovereignty, not matching Neuromancer XR's exact reasoning quality.

## 5. Multi-Agent Support

### 5.1 How Honcho Handles Multiple Agents

Honcho treats **every entity as a peer** — humans and agents alike. Each OpenClaw agent gets its own peer (`agent-{agentId}`). This means:
- Every agent has its own independent memory
- Agents can have peer cards (learned capabilities, behaviors)
- Agents can observe each other's messages in shared sessions

### 5.2 Peer Cards for Agent-to-Agent Memory

Peer cards are the mechanism for **agent-to-agent memory**. When Agent A interacts with Agent B, Agent B's peer card contains learned information about Agent A's capabilities, communication style, and state. This enables:

- Agents to understand what other agents can/cannot do
- Agents to know what other agents have already worked on
- Coordinated task handoffs without re-explaining context

### 5.3 OpenClaw Plugin: Subagent Hierarchy

The `@honcho-ai/openclaw-honcho` plugin has explicit multi-agent awareness:
- When OpenClaw spawns a subagent, the plugin tracks `parentPeerId` in metadata
- The spawning agent is added as a **silent observer** in the subagent's session (`observeMe: false, observeOthers: true`)
- This gives Honcho visibility into the full agent tree

### 5.4 Shared vs. Isolated Memory

| | DIY Karpathy | Honcho |
|-|-------------|--------|
| Shared memory | Shared filesystem (raw/ + wiki/) | Workspace-level, peer-specific |
| Agent isolation | Filesystem permissions | Peer-level isolation |
| Cross-agent reading | Can read any file | Can read peer cards of other agents |
| "Codebase as memory" | Native (just files) | Not designed for this |

**Key insight:** Honcho is not designed for "codebase as memory." It's designed for **conversation-driven user profiling**. For code and documentation, DIY Karpathy's filesystem approach is more appropriate.

## 6. Integration with Victor's Stack

### 6.1 OpenClaw (Native Plugin)

**Plugin:** `@honcho-ai/openclaw-honcho`  
**Installation:** `openclaw plugins install @honcho-ai/openclaw-honcho`  
**Setup:** `openclaw honcho setup`

The plugin:
- Replaces built-in memory plugins (memory-core, memory-lancedb) via the `memory` slot system
- Automatically migrates legacy files (USER.md, MEMORY.md, memory/, canvas/) to Honcho
- Provides 5 tools: `honcho_context`, `honcho_search_conclusions`, `honcho_search_messages`, `honcho_session`, `honcho_ask`
- Observes both user and agent messages after every AI turn
- Strips platform metadata before saving (only meaningful content persists)
- Configurable noise pattern filtering (heartbeats, cron boilerplate, etc.)
- Multi-agent aware (subagent hierarchy tracked via `parentPeerId`)

**Config options:**
```json
{
  "plugins": {
    "entries": {
      "openclaw-honcho": {
        "config": {
          "apiKey": "honcho_api_key",
          "workspaceId": "openclaw",
          "baseUrl": "https://api.honcho.dev",
          "noisePatterns": [],
          "disableDefaultNoisePatterns": false,
          "ownerObserveOthers": false
        }
      }
    }
  }
}
```

**For self-hosted Honcho:**
```json
{
  "plugins": {
    "entries": {
      "openclaw-honcho": {
        "config": {
          "apiKey": "",  // blank for local
          "baseUrl": "http://localhost:8000",
          "workspaceId": "openclaw"
        }
      }
    }
  }
}
```

### 6.2 Rails Integration

Honcho provides:
- **Python SDK:** `pip install honcho`
- **TypeScript SDK:** `npm install @honcho-ai/sdk`
- **REST API:** Any HTTP client works (Faraday, HTTParty, etc.)

No official Ruby SDK. For Rails, you'd use the REST API:

```ruby
require 'net/http'
require 'json'

class HonchoClient
  BASE_URL = ENV['HONCHO_BASE_URL'] || 'https://api.honcho.dev'
  API_KEY = ENV['HONCHO_API_KEY']

  def self.create_message(peer_id, session_id, content)
    uri = URI("#{BASE_URL}/v3/peers/#{peer_id}/sessions/#{session_id}/messages")
    Net::HTTP.post(uri, { content: content }.to_json, {
      'Authorization' => "Bearer #{API_KEY}",
      'Content-Type' => 'application/json'
    })
  end

  def self.get_context(peer_id, query)
    uri = URI("#{BASE_URL}/v3/chat")
    Net::HTTP.post(uri, { peerId: peer_id, message: { content: query } }.to_json, {
      'Authorization' => "Bearer #{API_KEY}",
      'Content-Type' => 'application/json'
    })
  end
end
```

### 6.3 n8n Integration

Honcho's REST API integrates easily with n8n:
- **HTTP Request node** for all API calls
- Webhook triggers for external events
- No special authentication beyond Bearer token

Key endpoints for n8n:
- `POST /v3/peers/{peerId}/sessions/{sessionId}/messages` — ingest messages
- `POST /v3/chat` — query user context
- `GET /v3/peers/{peerId}` — get peer representation

### 6.4 LLM Providers

Honcho is **provider-agnostic** for self-hosted. It supports:
- **vllm/custom slot** — any OpenAI-compatible API (OpenRouter, Venice, Together, Groq, Ollama)
- **openai** — OpenAI direct
- **anthropic** — Anthropic direct
- **google** — Google Gemini
- **groq** — Groq

For the OpenClaw plugin with managed Honcho cloud:
- Uses Plastic Labs' Neuromancer models (not configurable)

**Can it use OpenCode-Go? Kimi? MiniMax?**
- OpenCode-Go: If it has an OpenAI-compatible API endpoint, yes
- Kimi (Moonshot): Yes, via OpenAI-compatible API
- MiniMax: Yes, via OpenAI-compatible API

## 7. Honcho vs. DIY Karpathy

| Aspect | Honcho | DIY Karpathy |
|--------|--------|--------------|
| **Storage** | PostgreSQL + pgvector (structured DB) | Filesystem — `raw/` + `wiki/` directories |
| **Memory model** | Dialectic reasoning — formal logic extraction | Document/wiki synthesis — LLM reads and compiles |
| **Cross-session** | Built-in, automatic, async reasoning | Manual — `compile.py` runs LLM to synthesize |
| **Multi-agent** | Peer cards, agent-to-agent memory via shared workspace | Shared filesystem, agents can read each other's files |
| **Reasoning** | Background async workers extract conclusions proactively | On-demand, full-context reading during prompt assembly |
| **Setup** | ~3 min (community fork) to complex (manual) | Trivial — just mkdir |
| **Infrastructure** | PostgreSQL, Redis, Docker | Zero — just filesystem |
| **Visibility** | DB stores structured conclusions; harder to inspect | All files readable in text editor |
| **Cost** | Cloud (free tier / paid) or infra costs | Zero (files + API calls only) |
| **Neuromancer XR** | Available in cloud (86.9% LoCoMo) | N/A |
| **"Codebase as memory"** | Not designed for this | Native — just put files in raw/ |
| **Customization** | Configurable providers/models, but core logic opaque | Full control — you write every prompt |
| **Data portability** | SQL dump required | Plain text files — git-friendly |
| **Forgetting/editing** | Rewrite conclusions via reasoning | Edit text files directly |
| **Traceability** | Conclusions traced to premises (formal logic) | Less structured — depends on compile prompt |

### 7.1 Critical Comparison Points

**Visibility & Control:**
DIY Karpathy wins — every memory file is a plain text file you can read, edit, delete, or git-diff. Honcho stores structured conclusions in a database; inspecting them requires API calls or DB queries.

**"Codebase as Memory" Use Case:**
DIY Karpathy wins clearly. Honcho is designed for conversation memory, not code comprehension. It doesn't natively ingest code files and extract knowledge from them.

**Cross-Session Memory Quality:**
Honcho wins for conversational memory. The Deriver extracts explicit and deductive conclusions after every ~1000 tokens of messages, and Dream consolidates memories every 8 hours. DIY Karpathy depends on periodic compilation runs that synthesize raw messages into wiki entries.

**Setup & Maintenance Burden:**
DIY Karpathy wins on setup (trivial). But Honcho's managed cloud + OpenClaw plugin is also nearly zero-effort — just `openclaw plugins install @honcho-ai/openclaw-honcho` and `openclaw honcho setup`. The OpenClaw plugin even auto-migrates existing memory files.

**Cost:**
DIY Karpathy wins. No infrastructure costs. Honcho managed cloud has a free tier but paid beyond that. Self-hosting has infra costs.

## 8. Critical Questions Answered

### Q1: Does Honcho's dialectic reasoning work well for code/technical knowledge?

**Not really, for two reasons:**

1. **Design purpose:** Honcho is built for social reasoning — extracting preferences, beliefs, goals, and behavioral patterns from conversations. It's trained on social reasoning traces (Neuromancer XR: ~10,000 curated social reasoning traces). Code and technical knowledge are fundamentally different from social/dialectic knowledge.

2. **Code ingestion:** Honcho has no native code file ingestion. It processes messages. If you want to reason about a codebase, you'd need to stuff code into messages — which is awkward and lossy. DIY Karpathy's `raw/` folder accepts any file type and the compilation process is explicit.

**Verdict:** For code-heavy memory needs, DIY Karpathy wins decisively.

### Q2: How much setup effort for self-hosted Honcho vs. the value it provides?

**Honcho managed cloud + OpenClaw plugin: ~5 minutes**
```bash
openclaw plugins install @honcho-ai/openclaw-honcho
openclaw honcho setup  # prompts for API key, migrates files
openclaw gateway restart
```
This is nearly zero effort and handles everything automatically.

**Self-hosted (community fork): ~3-10 minutes**
- 3 minutes with the setup script (if Docker is installed)
- 10+ minutes if Docker needs installing and you want to configure LLM providers

**Self-hosted (official, manual): 30+ minutes**
- Requires understanding Docker, PostgreSQL, Redis, environment variables

**Value assessment:** For a single user on a VPS, self-hosting adds infrastructure complexity (PostgreSQL, Redis, Docker, monitoring, backups) without meaningful privacy benefit unless Victor has specific data sovereignty requirements.

### Q3: Is the community fork production-ready?

**Yes, for the Hermes Agent use case it was designed for.** The `elkimek/honcho-self-hosted` repo is well-documented, tested on Ubuntu 22.04, and includes:
- Hardened Docker configuration (non-root DB user, localhost-only ports)
- Primary + backup LLM provider failovers
- Prometheus/Grafana removed (simpler ops)
- Clear setup script with guided configuration

**Caveats:**
- Not officially maintained by Plastic Labs
- Designed around Hermes Agent's specific config format
- If upstream Honcho changes significantly, the overlay approach may break
- No official support channel

### Q4: Can Honcho handle the "codebase as memory" use case?

**No, not natively.** This is a fundamental architectural mismatch. Honcho's memory model is:
- **Conversation-driven** — messages as the input
- **Social reasoning** — extracting beliefs, preferences, goals
- **Peer-centric** — modeling entities, not documents

DIY Karpathy's approach:
- **File-driven** — any file type in `raw/`
- **Compilation-based** — LLM synthesizes understanding during compile
- **Document-centric** — wiki entries about anything

For a developer who wants "read my codebase and remember how things work," Honcho is not the right tool. DIY Karpathy (or tools like this-space/continue (CodeR) or ZerePy) are purpose-built for this.

## 9. When Honcho Wins vs. DIY Karpathy

### Honcho wins when:

✅ **Multi-user or multi-agent scenarios** — Each agent/user has a peer with rich cross-session memory, and peer cards enable agent-to-agent understanding. DIY Karpathy's shared filesystem doesn't model agents as first-class entities.

✅ **Conversation-heavy memory needs** — If the primary memory source is chat conversations (not code/docs), Honcho's dialectic reasoning extracts much richer insights than DIY Karpathy's compilation approach.

✅ **Zero infrastructure preference** — Managed Honcho cloud via the OpenClaw plugin requires zero infrastructure setup. DIY Karpathy requires maintaining the filesystem + compile pipeline.

✅ **Automated user profiling** — Honcho's Deriver automatically extracts conclusions without manual curation. DIY Karpathy requires you to write wiki entries manually or trust the compile process.

✅ **Want Neuromancer XR quality reasoning** — The managed cloud uses Neuromancer XR (86.9% LoCoMo), which significantly outperforms general-purpose models on social reasoning benchmarks.

### DIY Karpathy wins when:

✅ **Single user, code-heavy** — Victor's use case. Filesystem is native for code, docs, and any file type. Visibility is total.

✅ **Total visibility and control** — Every memory file is readable, editable, deletable. No DB opacity. Git-diff friendly. No "black box" reasoning.

✅ **Zero infrastructure** — No Docker, no PostgreSQL, no Redis, no managed service. Just files and API calls.

✅ **"Codebase as memory"** — Native. Just drop files in `raw/`. DIY Karpathy is designed for this.

✅ **Cost-sensitive** — Zero infrastructure costs.

✅ **Privacy via simplicity** — If you don't want your data on anyone's servers (including Plastic Labs'), the simplest privacy is not having a server at all.

## 10. Final Recommendation

### Recommendation: **Try Honcho managed cloud via OpenClaw plugin first, fall back to DIY Karpathy if needed**

**Rationale:**

The OpenClaw plugin makes Honcho **nearly zero-effort** to try:
```bash
openclaw plugins install @honcho-ai/openclaw-honcho
openclaw honcho setup  # enter API key, auto-migrate files
openclaw gateway restart
```

This is comparable setup effort to DIY Karpathy, but with:
- Automated dialectic reasoning instead of manual compilation
- Built-in cross-session memory (no `compile.py` to run)
- Structured, queryable conclusions
- Potential for richer user profiling

**The managed cloud free tier** is sufficient for personal use. Your conversation data goes to Plastic Labs' servers, but the plugin is built by Honcho's own team and actively maintained.

**If Honcho works well for your conversation memory needs**, keep it — the automated reasoning quality (especially Neuromancer XR) is genuinely better than DIY compilation.

**If Honcho doesn't fit** (e.g., code memory feels awkward, you want more visibility, cost concerns), **DIY Karpathy is still the right call**. The filesystem approach is fundamentally better for code and documentation.

**Do NOT self-host Honcho** unless you have specific data sovereignty requirements. The infrastructure complexity (PostgreSQL, Redis, Docker, monitoring, backups) adds meaningful operational burden for no practical benefit in a single-user scenario.

### Decision Matrix

| If you want... | Use... |
|----------------|--------|
| Best conversation memory quality | Honcho cloud + Neuromancer XR |
| Easiest setup (zero infra) | Honcho cloud via OpenClaw plugin |
| Best code/docs memory | DIY Karpathy |
| Most visibility/control | DIY Karpathy |
| Multi-agent memory sharing | Honcho (peer cards) |
| Zero infrastructure | DIY Karpathy |
| Data sovereignty | Self-host Honcho (complex) or DIY Karpathy (simple) |
| Automated reasoning | Honcho (async workers) |
| Plain text, git-friendly | DIY Karpathy |

#   git clone https://github.com/plastic-labs/honcho.git ~/honcho

services:

      - .env

      - CACHE_ENABLED=true
      - METRICS_ENABLED=false
    restart: unless-stopped

    image: pgvector/pgvector:pg15
    restart: unless-stopped
    ports:
      - "127.0.0.1:5432:5432"  # localhost only for security
