# Memoria Background Jobs — Detailed Implementation Plan (BullMQ)

## Context
Memoria is a Node.js/Express API + React UI for document ingestion, chunking, embedding, and semantic search. Currently ingest is synchronous. We need to make it async with background jobs and real-time progress.

**Tech stack:** Node 22, Express 5, TypeScript, Drizzle ORM, PostgreSQL 16 + pgvector, React 19 + Vite + Tailwind, Docker Compose.

## 3. Job Data Model & File Handling (Critical Detail)

**The Problem:** Multer `memoryStorage` gives us a `Buffer` in memory. With async jobs, we cannot pass the buffer directly to BullMQ (serialization issues, Redis 512MB string limit). We must persist the file to disk **before** enqueue, then pass the file path to the job.

**Solution — Switch Multer to Disk Storage:**
```typescript
const storage = multer.diskStorage({
  destination: (req, file, cb) => {
    const uploadDir = path.join(process.env.DATA_DIR || './data', 'uploads');
    fs.mkdirSync(uploadDir, { recursive: true });
    cb(null, uploadDir);
  },
  filename: (req, file, cb) => {
    cb(null, `${uuidv4()}-${file.originalname}`);
  }
});
const upload = multer({ storage });
```

**Cleanup policy:** Worker deletes the uploaded temp file after processing (success or failure).

**Job Data Payload (what goes into BullMQ):**
```typescript
interface IngestJobData {
  // For file uploads:
  filePath?: string;        // Absolute path to uploaded temp file
  fileName: string;         // Original filename
  // For web URLs:
  sourceUrl?: string;
  // Common:
  sourceType: 'pdf' | 'md' | 'text' | 'web';
  llmProcess?: boolean;     // Whether to LLM-reformat
  sourceId: string;         // Pre-generated UUID (created in DB as "pending")
}
```

**Pre-create Source Record:**
Before enqueueing, create a source row in DB with `status = 'pending'`. The worker updates this to `processing` → `completed`/`failed`. This gives us:
- Immediate source listing even before processing
- Ability to link jobs to sources
- Query jobs by source without relying solely on BullMQ

## 5. BullMQ Queue & Worker Design

### Queue: `ingest`
```typescript
// src/lib/queue.ts
import { Queue } from 'bullmq';

export const ingestQueue = new Queue('ingest', {
  connection: redisConnection,
  defaultJobOptions: {
    attempts: 3,
    backoff: { type: 'exponential', delay: 5000 },
    removeOnComplete: { count: 100 },   // Keep last 100 completed
    removeOnFail: { count: 50 },        // Keep last 50 failed
  }
});
```

### Worker: `ingestWorker`
```typescript
// src/workers/ingestWorker.ts
import { Worker } from 'bullmq';

const ingestWorker = new Worker('ingest', async (job) => {
  const { sourceId, filePath, sourceUrl, sourceType, llmProcess } = job.data;
  
  // Update source + job status
  await updateSourceStatus(sourceId, 'processing');
  
  try {
    // Step 1: Extract (10%)
    await job.updateProgress({ step: 'extract', percent: 10 });
    await updateJobProgress(job.id as string, 'extract', 10);
    
    const content = sourceType === 'web'
      ? '' // extractor fetches URL internally
      : await fs.readFile(filePath);
    const extracted = await extract(sourceType, content, sourceUrl || filePath);
    
    // Step 2: LLM Reformat (30%)
    await job.updateProgress({ step: 'reformat', percent: 30 });
    await updateJobProgress(job.id as string, 'reformat', 30);
    
    let processedText: string | undefined;
    const shouldProcess = llmProcess || (llmProcess === undefined && sourceType === 'pdf');
    if (shouldProcess) {
      processedText = await reformatToMarkdown(extracted.text);
    }
    
    // Step 3: Write raw source files (40%)
    await job.updateProgress({ step: 'store_raw', percent: 40 });
    await updateJobProgress(job.id as string, 'store_raw', 40);
    
    const { rawPath, processedPath } = await writeRawSource(sourceId, extracted.text, processedText);
    await updateSourcePaths(sourceId, rawPath, processedPath);
    
    // Step 4: Chunk (60%)
    await job.updateProgress({ step: 'chunk', percent: 60 });
    await updateJobProgress(job.id as string, 'chunk', 60);
    
    const rawChunks = splitIntoChunks(extracted.text, 400, 60);
    const processedChunks = processedText ? splitIntoChunks(processedText, 400, 60) : [];
    
    // Step 5: Embed (80%)
    await job.updateProgress({ step: 'embed', percent: 80 });
    await updateJobProgress(job.id as string, 'embed', 80);
    
    const rawEmbeddings = await embedBatch(rawChunks.map(c => c.text));
    const processedEmbeddings = processedChunks.length > 0
      ? await embedBatch(processedChunks.map(c => c.text))
      : [];
    
    // Step 6: Store chunks (95%)
    await job.updateProgress({ step: 'store', percent: 95 });
    await updateJobProgress(job.id as string, 'store', 95);
    
    // Store raw chunks
    for (let i = 0; i < rawChunks.length; i++) {
      const chunkPath = await writeChunkFile(sourceId, i, rawChunks.length, rawChunks[i].text, sourceType, sourceUrl || filePath);
      await createChunk({ sourceId, chunkIndex: i, totalChunks: rawChunks.length, chunkPath, content: rawChunks[i].text, embedding: rawEmbeddings[i].values, isProcessed: false });
    }
    
    // Store processed chunks
    for (let i = 0; i < processedChunks.length; i++) {
      const chunkPath = await writeChunkFile(sourceId, i, processedChunks.length, processedChunks[i].text, sourceType, sourceUrl || filePath);
      await createChunk({ sourceId, chunkIndex: i, totalChunks: processedChunks.length, chunkPath, content: processedChunks[i].text, embedding: processedEmbeddings[i].values, isProcessed: true });
    }
    
    // Complete (100%)
    await job.updateProgress({ step: 'complete', percent: 100 });
    await updateJobStatus(job.id as string, 'completed', 100, 'complete');
    await updateSourceStatus(sourceId, 'completed');
    
    // Cleanup temp file
    if (filePath) {
      await fs.unlink(filePath).catch(() => {});
    }
    
    return { sourceId, rawChunks: rawChunks.length, processedChunks: processedChunks.length };
    
  } catch (error) {
    // Cleanup temp file on failure too
    if (filePath) {
      await fs.unlink(filePath).catch(() => {});
    }
    await updateSourceStatus(sourceId, 'failed');
    await updateJobStatus(job.id as string, 'failed', undefined, undefined, error instanceof Error ? error.message : 'Unknown error');
    throw error; // Let BullMQ handle retry
  }
}, { connection: redisConnection, concurrency: 1 });
```

**Worker lifecycle:**
- Worker starts when the app starts (in `index.ts`, before `app.listen`)
- On SIGTERM/SIGINT, close worker gracefully: `await worker.close()`

## 6. WebSocket Architecture (Socket.IO)

### Why Socket.IO over native WS
- Room-based broadcasting (client joins room for specific job)
- Automatic reconnection
- Fallback to HTTP long-polling
- TypeScript support

### Server Setup
```typescript
// src/lib/socket.ts
import { Server } from 'socket.io';
import { createServer } from 'http';

export function setupSocketIO(httpServer: ReturnType<typeof createServer>) {
  const io = new Server(httpServer, {
    cors: { origin: '*' } // Internal only, safe
  });
  
  io.on('connection', (socket) => {
    console.log('Client connected:', socket.id);
    
    socket.on('subscribe:job', (jobId: string) => {
      socket.join(`job:${jobId}`);
    });
    
    socket.on('disconnect', () => {
      console.log('Client disconnected:', socket.id);
    });
  });
  
  return io;
}
```

### Integrate with BullMQ Events
```typescript
// src/lib/queueEvents.ts
import { QueueEvents } from 'bullmq';
import type { Server as SocketIOServer } from 'socket.io';

export function setupQueueEvents(io: SocketIOServer) {
  const queueEvents = new QueueEvents('ingest', { connection: redisConnection });
  
  queueEvents.on('progress', ({ jobId, data }) => {
    io.to(`job:${jobId}`).emit('job:progress', { jobId, ...data });
  });
  
  queueEvents.on('completed', ({ jobId, returnvalue }) => {
    io.to(`job:${jobId}`).emit('job:completed', { jobId, ...returnvalue });
  });
  
  queueEvents.on('failed', ({ jobId, failedReason }) => {
    io.to(`job:${jobId}`).emit('job:failed', { jobId, error: failedReason });
  });
  
  return queueEvents;
}
```

### WebSocket Client Hook
```typescript
// ui/src/hooks/useJobSocket.ts
import { useEffect, useState } from 'react';
import { io } from 'socket.io-client';

export function useJobSocket(jobId: string | null) {
  const [progress, setProgress] = useState({ step: '', percent: 0 });
  const [status, setStatus] = useState<'pending' | 'processing' | 'completed' | 'failed'>('pending');
  const [error, setError] = useState<string | null>(null);
  const [sourceId, setSourceId] = useState<string | null>(null);
  
  useEffect(() => {
    if (!jobId) return;
    
    const socket = io(window.location.origin); // Same origin, auto-detects /socket.io path
    socket.emit('subscribe:job', jobId);
    
    socket.on('job:progress', (data) => {
      setProgress({ step: data.step, percent: data.percent });
      setStatus('processing');
    });
    
    socket.on('job:completed', (data) => {
      setStatus('completed');
      setProgress({ step: 'complete', percent: 100 });
      setSourceId(data.sourceId);
    });
    
    socket.on('job:failed', (data) => {
      setStatus('failed');
      setError(data.error);
    });
    
    return () => { socket.disconnect(); };
  }, [jobId]);
  
  return { progress, status, error, sourceId };
}
```

## 9. Error Handling & Edge Cases

### Worker Failures
- **Step failure:** If any step throws, BullMQ auto-retries (3 attempts with exponential backoff)
- **Source record stays in DB** with `status = 'failed'` for visibility
- **Job record** updated with error message
- **Temp file cleanup:** Always run on both success and failure paths (use `finally`-like pattern)
- **Partial success prevention:** Don't write chunks until ALL embeddings succeed. If embed fails, no DB writes occur (BullMQ retry starts from beginning)

### Queue Stalls
- BullMQ has built-in stall detection. Default is fine for MVP.
- If worker dies mid-job, BullMQ will retry it.

### Redis Connection Loss
- BullMQ auto-reconnects to Redis
- If Redis is down, `ingestQueue.add()` will throw — return 503 to client

### Duplicate Prevention (Optional)
- Before enqueue, check if a source with same `sourceUrl` and `status != 'failed'` exists
- If yes, return existing source ID instead of creating new job
- **Skip for MVP** — add later if needed

## 12. Key Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Queue backend | BullMQ + Redis | Production-ready, retries, progress API, dashboard potential |
| File handling | Multer diskStorage | Buffers can't be serialized to Redis; disk is reliable |
| Real-time transport | Socket.IO | Rooms for per-job subscription, auto-reconnect, TypeScript |
| Job persistence | BullMQ + DB jobs table | BullMQ for queue logic, DB for querying/filtering by source |
| Source pre-creation | Create DB row before enqueue | Allows listing sources immediately, links job to source |
| Worker concurrency | 1 (sequential) | MVP simplicity; increase later if throughput needed |
| Cleanup | Delete temp file in worker | Both success and failure paths |
| Progress granularity | 6 steps (10/30/40/60/80/95/100) | Fine enough for user feedback, not too chatty |
