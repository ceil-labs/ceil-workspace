## 2026-04-29 19:45 — Ingest Architecture Decision: Background Jobs

**Victor's ask:** Make ingest async with background jobs + WebSocket progress tracking.

**Current problem:** Ingest is synchronous. User must keep page open. Large PDFs take 10-20s. Closing tab cancels request, leaves partial data.

**Proposed architecture:**
```
POST /api/ingest → enqueue job → return jobId immediately (200ms)
Background worker processes: extract → LLM reformat → chunk → embed → store
WebSocket broadcasts: progress %, step names, errors
```

**New features needed:**
1. **Jobs/Queue page** (`/jobs`) — List all ingest jobs with status
2. **Job detail** (`/jobs/:id`) — Real-time progress via WebSocket
3. **Nav link** — "Queue" in top navigation
4. **Ingest page update** — After submit, show "View in Queue →" link (new tab)

**Database schema addition:**
```sql
CREATE TABLE jobs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_id UUID,
  status VARCHAR(20) NOT NULL, -- queued|running|completed|failed
  progress INT DEFAULT 0, -- 0-100
  current_step VARCHAR(50), -- extract|reformat|chunk|embed|store
  error TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

**WebSocket events:**
- `job:created` — New job added to queue
- `job:progress` — `{ jobId, step, percent }`
- `job:completed` — Done, includes sourceId link
- `job:failed` — Error details

**Worker implementation:**
- In-memory job queue for MVP (array)
- Single worker process (setInterval loop)
- Process jobs sequentially (one at a time)
- Each step updates progress in DB + emits WebSocket event
