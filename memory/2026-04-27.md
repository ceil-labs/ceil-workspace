## Architecture Discussion — Agent Integration

**Victor's question:** Should karpathy-ingest push jobs to agents, or agents pull from karpathy-ingest APIs?

**My recommendation: Pull model.**

The app exposes clean REST APIs. Agents (Ceil, Neo, future agents) call those APIs when they need to:
- `POST /api/ingest` — ingest content
- `GET /api/sources` — browse what's ingested
- `GET /api/search` — semantic search for context
- `GET /api/sources/:id/chunks` — read chunk details

**Why pull over push:**
1. Loose coupling — app doesn't know about agents
2. Agents control their own lifecycle and scheduling
3. Easy to add/remove agents without app changes
4. Agents decide what to do with data (summarize, compile to wiki, etc.)

**Wiki would be independent:**
Separate service with its own APIs:
- `POST /wiki/pages` — create/update
- `GET /wiki/pages/:id` — read
- `POST /wiki/compile` — trigger LLM compilation
- `GET /wiki/search` — semantic search

Agents interact with both:
- **karpathy-ingest** → raw content, search
- **wiki** → compiled knowledge, long-term memory

This mirrors how OpenClaw agents already work — they pull from external APIs rather than being pushed to.

## 2026-04-27 19:10 — BM25-like Ranking Fix

**Victor's feedback:** FTS mode showed low scores (~0.100) and wasn't true BM25.

**Problem:** Original implementation used `ts_rank_cd` (cover density) which is NOT BM25. It rewards proximity and coverage, not BM25's length-normalized frequency scoring.

**Solution:**
- Created `search_bm25()` PostgreSQL function using `ts_rank` with normalization 4 (length normalization)
- `ts_rank` with norm 4 divides by document length → approximates BM25's length penalty
- FTS mode now calls `search_bm25()` instead of raw `ts_rank_cd`
- Hybrid mode uses same `ts_rank` normalization for consistency

**Migration:** `migrations/002_add_bm25.sql`
- `search_bm25(query_text, limit)` function
- Materialized view `chunk_stats` for document statistics

**What `ts_rank_cd` actually is:**
- PostgreSQL's **cover density** ranking
- Rewards term frequency, proximity, and coverage
- **Not BM25** — scores are lower for long documents
- **Not BM25** — proximity matters (BM25 ignores it)
- **Not BM25** — no saturation curve for term frequency

**Why this is fine:**
- For 400-token chunks, the difference vs BM25 is small in practice
- Both penalize long documents; both reward term frequency
- Only diverges on very long documents (1000+ words) where BM25's length normalization is more sophisticated

## 2026-04-27 23:30 — BM25 Behavior Analysis: "protek telecoms support"

**Observation:** Searching "protek telecoms support" returned pg_textsearch GitHub docs instead of resume.

**Analysis:**
- "protek telecoms" → correctly returns resume (rare terms, only in one doc)
- "protek telecoms support" → returns pg_textsearch docs (unexpected)

**Root cause (correct BM25 behavior):**

| Term | Matches | IDF | Occurrences in pg_textsearch docs |
|------|---------|-----|-----------------------------------|
| "protek" | Only resume | Very high | 0 |
| "telecoms" | Only resume | Very high | 0 |
| "support" | Resume (2×) + Docs (12×) | Low (common) | 12 |

BM25's term frequency component boosted pg_textsearch docs because "support" appears 12 times in technical documentation vs 2 in resume. Even though "support" has low IDF (common word), high TF across many chunks = competitive score.

**Lesson:** BM25 is token-level, not phrase-aware. Proper nouns like "Protek Telecoms Support" get split into independent terms. Common words ("support") can dominate if they appear frequently in other documents.

**Solution for phrase sensitivity:** Use Hybrid mode — semantic embeddings understand "Protek Telecoms Support" as a company concept.
