# 2026-04-04 — Agent Platform Deployment & Dev Environment Fixes

Production GHCR deployment working. Local dev environment fixed. Redis/solid_cache migration completed. n8n configured with single-port proxy. Key insight: memory grows from both what you save AND what you ask (bidirectional growth concept documented).

## What We Did

### 1. Production Docker Deployment (GHCR) ✅
- **Problem**: `solid_cache_entries` table missing, multi-database setup too complex
- **Solution**: Migrated to Redis for cache + Action Cable, single database
  - Added `redis` gem to Gemfile
  - Updated `production.rb`: Redis cache store, inline jobs during precompile
  - Updated `cable.yml`: Redis adapter instead of solid_cable
  - Simplified `database.yml`: single database (removed cache/queue/cable DBs)
  - Updated `docker-compose.ghcr.yml`: added REDIS_URL env var
- **Result**: Clean production deploy with `docker compose pull && up -d`

### 2. Local Dev Environment Fixes ✅
- **Problem**: `EACCES: permission denied` on `/app/node_modules/.yarn-integrity`
- **Root cause**: Named volume created with root ownership, doesn't match host UID/GID
- **Solution**: 
  - Updated `Dockerfile.dev`: create `appuser` with matching UID/GID
  - Added build args `USER_ID` and `GROUP_ID` in docker-compose.yml
  - Pre-create `node_modules` directory with correct ownership
- **Result**: Local dev `docker compose up` works seamlessly

### 3. Build Pipeline Understanding
- **Two independent builds**:
  - **Local dev**: `Dockerfile.dev` → builds on your machine
  - **Production**: `Dockerfile` → built by GitHub Actions, pushed to GHCR
- **Production workflow**: `git push` → Actions builds → `docker compose pull` on VPS

### 4. CI/CD Maintenance
- Updated GitHub Actions to `actions/cache@v5` and `actions/upload-artifact@v7`
- Fixed branch name from `main` to `master`
- Disabled system-test job (no browser tests yet)

## 6. Memory Architecture Insight — Bidirectional Growth 💡

**Key insight from Karpathy tweet discussion:**

> Memory grows from both sides: what you SAVE and what you ASK.

**Original architecture gap**: Only captured explicit inputs (papers, bookmarks). Missing implicit inputs (user questions).

**Solution**: Add **Query Reflection** routine as 5th maintenance worker:
- After each user query, extract topic
- If topic not in wiki → create "interest stub"
- If topic exists → increment query count
- Over time: stubs reveal what user cares about

**Directory addition**:
```
wiki/interests/              # NEW
├── rafconsensus.md          # Interest stub: queried 5 times
└── distributed-systems.md   # Interest stub: queried 3 times
```

**Why this matters**: Your questions reveal knowledge gaps. Tracking them shows what needs documentation. High query count = high priority for research.

**Status**: Documented in `karpathy-architecture-design.md`. Implementation pending.
