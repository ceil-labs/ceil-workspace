## 2026-04-26 20:00 — Search Bug Investigation (Three Stacked Bugs)

**Context:** Victor tested semantic search with queries like "field service management", "PhD industrial engineering", "coffee roastery ERP" — all returned empty `[]` despite embeddings existing in DB.

### Bug 1: `db` Proxy Lost `this` Context
**File:** `src/db/client.ts`  
The `db` export was a Proxy that forwarded to `getDb()`, but `db.execute()` lost its `this` binding. Fixed by importing `getDb` and calling it directly in routes.

### Bug 2: IVFFlat Index with 100 Lists on 7 Rows
**File:** `src/db/migrations/0001_pgvector.sql`  
Migration created: `CREATE INDEX idx_chunks_embedding ON chunks USING ivfflat (embedding vector_cosine_ops) WITH (lists='100')`  
With only 7 rows, most of the 100 lists were empty → index returned nothing for `LIMIT` queries. Dropped the index. Sequential scan is faster for small tables anyway.

### Bug 3: Drizzle Parameterized Vector in `ORDER BY`
**File:** `src/routes/search.ts`  
Even with raw SQL, Drizzle's `sql` tag wrapped the vector string as a parameter (`$1`). Combined with `LIMIT` → 0 rows. Fixed by inlining the vector in `ORDER BY` via `sql.raw()`:
```ts
ORDER BY c.embedding <=> ${sql.raw(`'${vectorStr}'`)}::vector
```

### Result
Search now works. `"field service management"` returns 3 chunks with similarities 0.63, 0.62, 0.61.

## 2026-04-26 18:30 — Batch Embedding Content Object Fix

**Bug:** `batchEmbedContents` API requires `Content` objects with `parts: [{ text }]` structure, not plain strings. Was passing strings → API error.

**Fix:**
```ts
const batchRequest = {
  requests: texts.map((text) => ({
    content: { parts: [{ text }] },
    taskType: "SEMANTIC_SIMILARITY",
    outputDimensionality: 768,
  })),
};
```

This enables true batching: 10 texts in 1 API call instead of 10 sequential calls.

## 2026-04-26 22:15 — Google Image Generation Debugging Complete

**Problem:** `image_generate` calls with Google provider were aborting/timeout despite API key working for text.

**Root cause:** Google's `gemini-3.1-flash-image-preview` takes 60–120 seconds to generate images. OpenClaw's Google provider hardcodes a default timeout of `60s` (`timeoutMs: req.timeoutMs ?? 6e4`). When generation exceeds 60s, the request gets aborted with "This operation was aborted".

**Fix discovered through background curl testing:**
- Must pass `timeoutMs: 300000` (5 min) in the `image_generate` call parameters
- Must also specify `aspectRatio: "1:1"` for reliable generation (without it, API sometimes times out or returns no image)

**Working example:**
```json
{
  "action": "generate",
  "prompt": "A simple red circle on white background",
  "aspectRatio": "1:1",
  "timeoutMs": 300000
}
```

**Key findings from direct API testing:**
- API key works for text (`/v1beta/models/gemini-2.0-flash:generateContent`)
- Direct curl to image generation endpoint works but takes 90–120s
- Without `imageConfig` in payload → inconsistent results (sometimes timeout)
- With `imageConfig: {"aspectRatio": "1:1"}` → consistently works
- Response contains `inlineData.data` base64 JPEG — the provider code expects this format and it IS present

**Code inspection:** Read `dist/extensions/google/image-generation-provider.js` and `model-id.js` — provider correctly parses `inlineData` field. The issue is purely timeout-related.

**Config note:** `imageGenerationModel` is set in `agents.defaults` as `"google/gemini-3.1-flash-image-preview"`. The model ID normalizer maps `gemini-3.1-flash-image-preview` to `gemini-3-flash-preview` which is wrong for image generation, but the provider bypasses normalization for image models and uses the raw ID.

**No global timeout config available** — must pass `timeoutMs` per-call.
